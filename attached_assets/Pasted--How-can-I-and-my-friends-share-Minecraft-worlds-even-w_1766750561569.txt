
How can I and my friends share Minecraft worlds even when we're not all online so anyone can play regardless who's online?

25 Dec
You have a few good options for sharing a Minecraft world that stays accessible 24/7:

Dedicated Server (Best option) Run Minecraft server software on a computer that stays on all the time. This could be:

A spare computer at someone's house
A Raspberry Pi (though performance depends on how many players)
A rented cloud server (costs ~$5-15/month)
The server runs independently, so anyone can join anytime. You set it up once, then it just works.

Realm (Easiest option) Pay Mojang ~$8/month for an official hosted world. It's always online, backs up automatically, and requires zero technical setup. Just subscribe and invite friends.

Free hosting services Sites like Aternos or Minehut offer free servers, but they usually:

Have queues to start the server
Auto-shutdown after inactivity
May have performance limitations
How it works: Instead of one person hosting in-game (where closing Minecraft stops everyone), dedicated servers run separately. Players connect to an IP address/domain name, and the world persists between sessions.

For a technical person like you, I'd suggest trying the self-hosted server route first - you'd have full control and could run it on a Pi 5 if you have one spare. Want help setting that up?

How would a raspberry pi 4 hold up and how would we set this up?

25 Dec
Performance on Pi 4:

A Pi 4 can handle a small Minecraft Java server reasonably well:

2-5 players: Works fine with vanilla Minecraft
Render distance: Keep it around 8-10 chunks (not 32 like on gaming PCs)
Mods: Vanilla or lightweight mods only - heavy modpacks will struggle
RAM: Use the 4GB or 8GB model if possible
Performance gets choppy with many players or complex redstone contraptions, but for casual play with friends it's solid.

Setup Steps:

Install Raspberry Pi OS Lite (headless version saves resources)
Install Java:
bash
   sudo apt update
   sudo apt install default-jdk
Download server software:
bash
   mkdir minecraft-server
   cd minecraft-server
   wget https://papermc.io/api/v2/projects/paper/versions/1.20.4/builds/latest/downloads/paper-1.20.4.jar
(PaperMC is optimized for better performance than vanilla)

Run server first time:
bash
   java -Xmx2G -Xms1G -jar paper-1.20.4.jar nogui
This creates config files. Edit eula.txt and change false to true

Configure server.properties:
Set view-distance=8
Set your world name, difficulty, etc.
Run it again - server starts properly now
Port forwarding: Log into your router, forward port 25565 to your Pi's local IP
Share your public IP with friends (find it at whatismyip.com)
Auto-start on boot: Create a systemd service so it runs automatically when Pi restarts.

What size storage will I need?

25 Dec
Storage Requirements:

Minimal setup: ~2-3 GB

Server software: ~50-200 MB
Fresh world: ~100-500 MB initially
Operating system: ~2 GB
Realistic for actual use: 8-16 GB

As you explore, the world grows (Minecraft generates new chunks as players wander)
A moderately explored world with several players: 2-5 GB after a few weeks
Backups: You'll want space for at least a few backup copies
Growth rate: Expect roughly 50-200 MB per week of active play, depending on how much exploring happens. If players spread out far, it grows faster.

Recommendation:

Minimum: 16 GB SD card (leaves ~10 GB usable after OS)
Better: 32 GB SD card - gives you breathing room for backups and world growth
Best: 64+ GB if you want multiple worlds or extensive backups
Pro tip: Consider running the server from an external USB SSD instead of the SD card. It's faster (less lag) and more reliable for constant read/writes. SD cards can wear out with heavy server use. A small 128 GB USB SSD costs ~Â£15-20 and performs much better.

Your Pi 4 should handle this fine - the storage is likely your main constraint, not the hardware.

Auto-Start Minecraft Server with systemd
What is systemd?
systemd is Linux's system and service manager. It controls what programs run when your Pi boots up, monitors them, and can automatically restart them if they crash.

How It Works
When you create a systemd service for Minecraft:

On boot: Pi starts â†’ systemd starts â†’ Minecraft server starts automatically
While running: systemd monitors the server and can restart it if it crashes
Manual control: You can stop/start/disable it with simple commands
Logging: All server output goes to systemd logs (viewable with journalctl)
Advantages
âœ… Convenience
No manual startup needed - server runs automatically when Pi boots
Survives reboots - perfect for power outages or updates
No screen/SSH needed - server runs in background independently
âœ… Reliability
Auto-restart on crashes - if server crashes, systemd restarts it automatically
Proper shutdown on reboot - systemd stops server cleanly before Pi shuts down
Dependency management - ensures network is ready before starting
âœ… Management
Simple commands - sudo systemctl start minecraft instead of long Java commands
Status checking - see if server is running: sudo systemctl status minecraft
Easy disable - turn off auto-start without deleting anything
âœ… Professional Setup
Standard Linux practice - how production servers run services
Integrated logging - logs stored with system logs, easy to search
Resource control - can set CPU/memory limits if needed
Disadvantages
âŒ Less Direct Control
Can't see console directly - no immediate > prompt when SSH'ing in
Requires systemctl commands - can't just run the Java command
Stops on its own - if you're mid-game and Pi reboots, server starts before you're ready
âŒ Debugging Complexity
Log viewing is different - use journalctl instead of seeing output directly
Permission issues - runs as specific user, file permissions matter more
Startup errors hidden - if server fails to start, need to check logs
âŒ Resource Usage
Always running - uses RAM/CPU even when nobody's playing
Boot time - Pi takes longer to boot (waits for Minecraft to start)
Background drain - if you want Pi for other tasks, Minecraft is using resources
âŒ Update Friction
Can't update while running - must stop service first
File locking - systemd holds files, harder to move/rename things
Restart needed - config changes need service restart, not just server restart
When to Use Auto-Start
âœ… Good for:
Dedicated server - Pi only/mainly used for Minecraft
Regular play schedule - friends play daily/weekly at set times
24/7 server - want it always available
Remote/headless setup - Pi in closet, no monitor
Unreliable power - frequent outages, want server back up automatically
âŒ Not ideal for:
Occasional play - only use server sometimes
Multi-purpose Pi - using Pi for other projects too
Active development - frequently changing configs/plugins
Limited resources - Pi struggling with performance
Manual control preference - like starting server yourself

RAM Optimization
Current Setting

Xmx2G: Maximum 2GB RAM

Optimized Setting for Pi 4
bash
-Xmx2G -Xms2G
Why set them equal?

Prevents garbage collection overhead - Java won't waste time allocating more RAM
More predictable performance - no sudden lag spikes when RAM grows
Better for server - consistent memory footprint
Your Pi 4 has 4GB or 8GB total, so 2GB for Minecraft leaves plenty for the OS.

CPU/Java Optimizations
Garbage Collection (GC) Tuning
Java's garbage collector cleans up unused memory. The default GC can cause lag spikes. We can use Aikar's Flags - widely used optimized settings for Minecraft servers:
